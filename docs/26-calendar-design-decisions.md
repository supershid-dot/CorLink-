# 26 — Calendar Module: Design Decisions and Implementation Alignment

**Type:** Retrospective design-decision document, required by `docs/23-rooms-meetings-implementation-specification.md` Phase C ("This phase gets its own separate design-decision doc before any schema work begins, mirroring the `docs/09`/`docs/12` precedent, given it is a genuinely new module"). That document was not written before Calendar's frontend work began — Calendar shipped directly from `docs/22`/`docs/23`'s own specification instead. This document exists to close that gap after the fact: it records the architecture actually shipped, reconciles every point where the implementation differs from `docs/23`'s original Phase C text, and records what remains deferred.

## 1. Architecture shipped

The implementation deliberately departs from `docs/23` Phase C's proposed architecture on the single biggest question that phase posed — whether Calendar reads through a new composed RPC — and answers it using that same phase's own escape hatch:

- **Calendar is a read-only composition layer, not a new data source.** `CalendarAPI.fetchEvents({from, to})` (`js/data/calendar-api.js`) calls `MeetingsAPI.fetchMeetingsInRange()`, `RoomsAPI.fetchBookings()`, and `RoomsAPI.fetchRoomBlocks()` via `Promise.all()` and merges their results client-side into one normalized, sorted event array. No table is queried by this feature that isn't already queried elsewhere in this codebase.
- **No new RPC was implemented.** `docs/23` §4 specified `calendar_events_for_range(p_from, p_to, p_viewer_user_id)` as a purpose-built composed-read function. It was never built. §2 of that same Phase C spec explicitly permitted this outcome: *"a single composed read RPC is preferred over five separate client-side merges — decide only after measuring, not preemptively."* No measurement ever showed the client-side merge to be a problem, so the RPC was never justified and the simpler path shipped instead.
- **No new RLS policy was written.** Every event Calendar displays passes through the exact SELECT policy that already governs it directly — `meetings_select` (via `can_view_meeting()`), `meeting_room_bookings_select`, and the existing room-block SELECT rule — because Calendar's data layer literally calls the same `MeetingsAPI`/`RoomsAPI` functions the Meetings and Rooms views themselves call. This is the specific safeguard `docs/23` §3 required ("reuse each underlying table's existing, already-correct SELECT policy... rather than reimplementing visibility logic behind a new bypass"), satisfied by construction rather than by a new bypass that then has to be independently proven safe.
- **Client-side event normalization is the entire "composition" step.** `fetchEvents()` maps each of the three raw result sets into one shared shape (`{type, id, title, start, end, status, orgId, roomId, roomName, creatorId, creatorName, meetingType, isRecurring, isLocked, isDraft, locationMode, raw}`), then concatenates and sorts by `start`. This normalization is display-layer field renaming only — it contains no permission or business logic, and it cannot widen what any given row's own RLS already decided the caller may see.
- **Existing Meetings and Rooms APIs are reused verbatim, not duplicated.** `MeetingsAPI.fetchMeetingsInRange()` and `RoomsAPI.fetchRoomBlocks()` gained optional range parameters (additive, backward-compatible) so Calendar could call them directly; `RoomsAPI.fetchBookings({from, to})` already supported range filtering. No parallel "calendar-scoped" copy of any query exists.
- **Permission inheritance is total and automatic, not re-implemented.** Because every event Calendar renders is fetched through the same API call the source module (Meetings or Rooms) would make, every existing permission behavior — organization isolation, locked-meeting read-only state, private/participant/organization visibility tiers, room-manager-only fields — applies to Calendar with zero Calendar-specific permission code. Click-through reuses the existing `meetingId`/`bookingId` query-param navigation mechanism (already used by the notification dropdown), so opening an event from Calendar opens the exact same detail modal Meetings/Rooms would open, with the exact same authorization checks that modal already runs.
- **A `meeting_room_bookings` row with a non-null `meeting_id` is intentionally excluded from the "standalone booking" event type**, filtered via `.filter(b => !b.meeting_id)`. A room-booked meeting is already represented once by its own `meeting` event (which carries the booking's room via `MeetingsAPI.activeBooking(m)`); including the same booking again as a separate `booking` event would double-render one real-world occupancy as two calendar entries. This de-duplication rule was verified empirically during the Calendar regression review — no double-rendering occurs for room-booked meetings, recurring or otherwise.
- **Recurring meeting occurrences are rendered as ordinary meetings, with no special-case UI.** `docs/25` §1 already established that every occurrence is a normal `meetings` row (`series_id`/`series_occurrence_date` set, `can_view_meeting()`/`meetings_select` otherwise unaware of recurrence). Calendar's `meetingEvents` mapping sets `isRecurring: !!m.series_id` purely as a display flag (a small recurrence indicator, per `docs/22` §3.2's item-type table) — the event is fetched, filtered, permissioned, and click-routed identically to a non-recurring meeting. No occurrence-aware branch exists anywhere in `calendar-api.js` or `calendar.js` beyond that one boolean.

## 2. Currently supported event types

Of the six item types `docs/22` §3.2 specified for Calendar's "single, complete schedule view," four are implemented and two are deferred:

**Supported:**
- **Meetings** — every `meetings` row the caller can already see via `can_view_meeting()`, including `status='draft'` meetings (an existing, already-shipped lifecycle state — see §3 below on why this is not the same thing as the deferred "Draft/Pre-booked Meetings" feature).
- **Recurring meeting occurrences** — rendered as ordinary meeting events with a recurrence indicator; see §1 above.
- **Standalone room bookings** — `meeting_room_bookings` rows with no linked meeting.
- **Room blocks** — active room maintenance/block windows.

**Deferred (not built by this implementation):**
- **Draft / Pre-booked Meetings** — the `docs/23` §4 bulk-creation feature (`is_draft_series`, `days_of_week`, `recurrence_pattern='custom_days'`) does not exist anywhere in the codebase yet (per `docs/25` §1's own confirmation that it shipped neither in Recurring Meetings Phase 1 nor since). There is therefore nothing of this type for Calendar to source. See §3 below for the distinct, already-supported `meetings.status='draft'` state, which is not this feature.
- **Staff leave** — `docs/23` Phase H (self-service leave records, org-admin management, leave types, Calendar indicator) has not been implemented. No leave table, RLS, or RPC exists, so Calendar has no leave data source to compose.

Because both deferred types are simply absent data sources rather than partially-built ones, their eventual implementation is additive to `CalendarAPI.fetchEvents()` — a fourth and fifth `Promise.all()` member each, normalized into the same shared event shape — and requires no rework of anything shipped now.

## 3. `meetings.status='draft'` vs. Draft/Pre-booked Meetings — a deliberately preserved distinction

`meetings.status` already includes a real `'draft'` value as part of the existing meeting lifecycle (predating this Calendar work). This is a single meeting created but not yet finalized by its own creator — entirely different from `docs/23` §4's unshipped Draft/Pre-booked Meetings feature, which is a bulk-creation mechanism for placeholder slots across a date range. `CalendarAPI` flows the former through unchanged (`isDraft: m.status === 'draft'`) as a normal meeting event with a status flag; it does not, and could not, source the latter, since no code anywhere creates that kind of row. This distinction was carried through the implementation deliberately, in code comments and here, so a future reader does not mistake "Calendar already shows drafts" for "Draft/Pre-booked Meetings is done."

## 4. Current UI capabilities

`js/views/calendar.js` implements four view modes, all operating over the same normalized event set from `CalendarAPI.fetchEvents()`:

- **Day** — single-day event list.
- **Week** — seven-column grid, one column per day, events listed inside each column (not a positioned hour-grid — see §5).
- **Month** — fixed 42-cell grid (the Sunday on/before the 1st through 41 days later), each cell listing its day's events with overflow handled via a "+N more" link.
- **Agenda** — flat chronological list grouped by day.

Filtering is applied client-side, after fetch, over the already-permissioned event array: organization, room, creator, status, meeting type, and a self-scoped "only my meetings" toggle (via `MeetingsAPI.fetchMyMeetingIds()`, never an arbitrary-other-user picker — see §6 below on why the staff picker from `docs/23` §4 was not built this way). Every filter dropdown's options are derived by iterating the already-fetched events (`Map`/`Set` collection), never by a separate query, which guarantees a filter option can never expose something the caller couldn't already see.

## 5. Known deferred improvements

Identified during implementation and the subsequent regression review; none are correctness or security gaps, and none block the current shipped state:

- **Hour-grid layout.** Week and day views currently list events inside their day/column rather than positioning them against a vertical time axis. `docs/22` §3.2's "week-grid / day-agenda visual schedule" language anticipated a true positioned grid (the same visual language MeetFlow's Rooms tab uses); the shipped list-per-day layout satisfies the same information need with less rendering complexity, at the cost of not visually showing overlap/duration at a glance. A future pass can add real time-axis positioning without changing any data-layer code.
- **Organization-name cache.** `orgId` is resolved to a display name per-render (the caller's own cached profile covers the single-org case with zero extra query; a super admin viewing multiple organizations resolves the rest via `AdminAPI.listOrganizations()`, called fresh rather than cached). A small client-side cache would avoid repeat calls for multi-org super-admin use, but no performance problem has been observed to justify it yet.
- **Calendar performance indexes.** The regression review that preceded this document confirmed no database index currently supports Calendar's date-range queries against `meetings`, `meeting_room_bookings`, or `meeting_room_blocks` efficiently at scale (`EXPLAIN` showed a sequential scan in local testing). This was flagged, not fixed, during that review, per its review-only scope. No index has been added by any step since. This remains open for a future, measurement-driven indexing pass.
- **Calendar module dependency messaging.** `docs/22` §6 specified Calendar as sequenced strictly after Phase F (Recurring/Drafts) and Phase H (Leave) — "Do not build Calendar (Phase C) before Phase F and Phase H are done." This implementation shipped after Recurring Meetings Phase 1 but before Phase H (Leave) and before Draft/Pre-booked Meetings, deliberately narrowing scope to the four event types in §2 rather than waiting on both dependencies. Calendar's own module-enablement gate (`platform_modules.route`/`organization_modules.is_enabled` for the `calendar` key) is also fully independent of the `meetings`/`rooms` module gates — enabling Calendar for an organization that has not enabled Meetings or Rooms produces an empty schedule with no explanatory message, rather than a UI hint that those modules should be enabled too. This was confirmed, not fixed, during the regression review, and is recorded here as a deferred UX improvement, not a security issue (RLS/module-gating enforcement itself is unaffected — a user simply sees nothing to show, never something they shouldn't).

## 6. Implementation deviations from `docs/23` Phase C

- **No composed RPC** (`calendar_events_for_range`) — see §1. Explicitly permitted by `docs/23` §2's own "decide only after measuring" language.
- **No `who_can_i_view_schedule_for()` staff-picker RPC, and no staff picker UI.** `docs/23` §4/§8 specified a from-scratch replacement for MeetFlow's broken per-viewer-grant mechanism, letting a supervisor/admin view another named staff member's schedule. This was not built. Calendar's only participant-scoped filter is "only my meetings," self-scoped via the caller's own already-correct `fetchMyMeetingIds()` RLS — deliberately not attempting an arbitrary-other-user query, because `meeting_participants_select`'s RLS (own row, or a meeting one manages) would silently return an *incomplete* result set for "show me user X's meetings" whenever X isn't the caller and the caller doesn't manage every one of X's meetings. This is a scope reduction, not a broken implementation of the original spec — no privilege-escalation-shaped mechanism (the specific risk `docs/22` §3.2 called out from MeetFlow's `ssa_viewer_<id>` pattern) exists in this codebase to be reused, replicated, or misused.
- **No `.ics` export.** `docs/22` §3.2 and `docs/23` §5 both listed calendar export as a capability; it was not implemented in this round.
- **No section-based color coding.** Listed in `docs/22` §3.2's capability table; not implemented — event styling is currently by event *type* (meeting/booking/block, plus a locked/draft/cancelled variant for meetings), not by section.
- **Leave and Draft/Pre-booked Meetings are absent, not partially built** — see §2 and §3.
- **Shipped ahead of `docs/22` §6's stated hard sequencing** (Phase C after Phase F *and* Phase H) — see §5's "Calendar module dependency messaging" entry. Recurring Meetings Phase 1 (Phase F) had shipped; Phase H (Leave) had not. This was a deliberate scope narrowing (four event types instead of six) rather than an oversight, and is recorded here so the deviation from the roadmap's stated ordering is a documented decision, not something a future reader has to notice on their own.
- **Migration footprint reduced to a single one-line route-activation patch.** `docs/23` §9 anticipated `supabase/patch-calendar-module-foundation.sql` (new RPCs, and — per the roadmap's earlier phasing note — a module-key seed row) followed by a separate `supabase/patch-calendar-route-activation.sql`. In practice, the `calendar` module key was already seeded with `route = NULL` by the original `patch-platform-module-foundation.sql`, before this implementation began, and no RPC was ever built (see above) — so `patch-calendar-route-activation.sql`'s one-line `UPDATE platform_modules SET route = 'calendar' ...` flip was the *only* SQL this implementation needed. No foundation migration was written because there was nothing left for one to do.

## 7. Relationship to `docs/22` and `docs/23`

This document does not supersede `docs/22`/`docs/23` — it is the retrospective decision record those documents' own process (Phase C §"gets its own separate design-decision doc") required, reconciling their prospective specification against what was actually built. `docs/22` §3.2/§6 and `docs/23` Phase C are both updated by this same documentation step to point back here and to record Calendar's shipped status, its client-side composition architecture, and its deferred capabilities.

## 8. Draft rendering is now backed by a fully hardened draft lifecycle

At the time this document was written, `meetings.status='draft'` was a real but comparatively thin lifecycle state — Calendar already rendered it correctly (§2/§3 above), but several of the RPCs a draft meeting could still reach (RSVP, attendance, minutes, locking, cancellation) had not yet been made draft-aware, and participant-facing notifications were not yet suppressed while a meeting remained a draft. `supabase/patch-meetings-drafts.sql` (committed `b6ee08c`, after this document) closed those gaps — see `docs/27-draft-meetings-design-decisions.md` for the full record. Calendar's own code required no change: `CalendarAPI.fetchMeetingsInRange()` was already status-unfiltered and `calendar.js`'s `calendar-event--draft` styling already existed, so the draft events Calendar has always shown are now backed by a complete, hardened draft lifecycle rather than a bare status flag. §3's distinction between this state and the still-unbuilt bulk Draft/Pre-booked Meetings mechanism remains accurate and unchanged by this update.
