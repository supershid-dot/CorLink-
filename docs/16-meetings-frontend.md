# Meetings Frontend

**Type:** Implementation record for the Phase 3 frontend (`docs/03-migration-architecture.md` §8), built directly against the already-implemented and already-tested database layer (`supabase/patch-meetings-foundation.sql`, `docs/14-meetings-database-foundation.md`). No new database objects were designed in this step — only the exact objects that migration already ships.
**Files:** `js/data/meetings-api.js`, `js/views/meetings.js`, `supabase/patch-meetings-route-activation.sql`, plus surgical edits to `index.html`, `js/app.js`, `js/router.js`, `js/views/shell.js`, `js/views/rooms.js`.
**Date:** 2026-07-22
**Scope actually applied:** Local, disposable PostgreSQL instance only (`corlink_meetings_local`, already carrying both the Rooms/Booking and Meetings migrations from prior sessions). **Neither the CorLink nor the MeetFlow hosted Supabase project was accessed or modified.** MeetFlow was not touched. Rooms/Booking's own workflows were not modified — the one integration touch (below) is additive and does not change any Rooms/Booking behavior. Nothing was pushed to any remote branch.

## 1. What was built

### Route and navigation

- `#meetings` — a single route, four client-side tabs (Upcoming, My Meetings, Past, Cancelled). Mirrors `rooms.js`'s single-route/multi-tab shape rather than `requests.js`'s separate list/detail route pair, since there is no natural "detail page" distinct from a meeting-details modal reachable from any tab — Meetings has no approval-workflow tab of any kind (no such concept exists in this domain), so unlike Rooms's manager-only Pending Approvals tab, all four Meetings tabs are unconditionally shown to every user with module access.
- `js/router.js`'s `MODULE_ROUTES['meetings'] = 'meetings'` — direct-URL and nav-link access both gated through the existing `moduleGuardPasses()`/`isModuleEnabled()` machinery, fail-closed exactly like every other module route. No new gating mechanism was introduced. `layer2Allows()`'s default `return true` branch applies (any active org member with the module enabled may reach the route), matching `docs/12` §12's "any active staff member may create a meeting" and mirroring `rooms`/`requests`/`entry`'s identical Layer 2 shape.
- `js/views/shell.js` — a `Meetings` link added to the sidebar, topbar, and mobile bottom-nav, each gated by `AppShell.isModuleEnabled(user, 'meetings')`, matching the existing `showRooms`/`showEntry` pattern exactly.
- `js/app.js` — `Router.register('meetings', MeetingsView)`.

### `supabase/patch-meetings-route-activation.sql` (new, small, additive)

`platform_modules.route` for `module_key = 'meetings'` was left `NULL` by `patch-meetings-foundation.sql` (`docs/12` §19 step 3: "stays `NULL` until [the frontend] ships"). `admin.js`'s Modules tab reads this column directly to decide whether an org's Enable toggle is clickable at all — without flipping it, no organization admin could ever turn Meetings on for their org. This migration is a single idempotent `UPDATE ... WHERE route IS DISTINCT FROM 'meetings'`, wrapped in `BEGIN`/`COMMIT`, touching no table, RLS policy, trigger, or function, and scoped to the `'meetings'` module key only — it does not touch `'rooms'` or any other module. Verified locally: applied cleanly (`UPDATE 1`), re-ran a second time and confirmed zero rows affected (`UPDATE 0`) with the value unchanged.

### `js/data/meetings-api.js`

An IIFE data-layer module, same shape as `rooms-api.js`. `meetings`/`meeting_participants` carry SELECT-only RLS with zero write policy for any role (`docs/12` §18) — every mutation on either table is a bare `db.rpc(name, params)` passthrough using the RPCs' **exact** parameter names, verified directly against the live `corlink_meetings_local` schema (function signatures, not recalled from any earlier design document). No `.insert()`/`.update()`/`.delete()` call against `meetings` or `meeting_participants` exists anywhere in this file — confirmed by grep on the finished file.

Required functions, all present with the exact requested names: `fetchMeetings`, `fetchMeeting`, `fetchMeetingParticipants`, `createMeeting`, `updateMeeting`, `cancelMeeting`, `addParticipant`, `removeParticipant`, `assignRoomBooking`, `detachRoomBooking`. Small, necessary additions beyond that exact list: `fetchMyMeetingIds`/`fetchMyMeetings` (the "My Meetings" tab's creator-or-organizer-or-participant query, §6 below), `fetchLinkedBooking` (a focused single-row read of a meeting's active room booking, used by the detail modal), and `activeBooking(meeting)` (a small, pure, in-memory helper that picks the one active booking out of a meeting's embedded `bookings` array — no network call).

**Participant contact display always goes through `meeting_participant_list()`**, never a raw-table read of `meeting_participants` for anyone else's row — the one place this file reads the raw table directly (`fetchMyMeetingIds`) only ever reads the caller's **own** `user_id = auth.uid()` row (permitted by `meeting_participants_select`'s own narrower policy), never another participant's contact fields.

Actor identity for every RPC comes from `auth.uid()` server-side; this file never sends a client-supplied user id as "who did this" — `add_participant`'s `p_user_id` parameter is the **target** internal user being added, not an actor override, and is handled identically to every other target-identity parameter already used throughout this codebase (e.g. `RoomsAPI.addRoomManager`'s `userId`).

**`bookings` list embed, not per-row N+1:** `MEETING_SELECT` embeds every `meeting_room_bookings` row ever linked to a meeting (`bookings:meeting_room_bookings!meeting_room_bookings_meeting_id_fkey(...)`), including cancelled/rejected history, rather than issuing one query per meeting to find its active booking. `activeBooking()` then picks the one active (`hold`/`pending`/`confirmed`) row client-side. This one-to-many embed pattern is the same shape already shipped and working in this exact codebase (`requests-api.js`'s `responses(id, status, ...)` embed) — confirmed by reading that file directly before writing this one, since a live PostgREST instance was not available this session to execute the query itself (see §3 below for what "confirmed" precisely means here).

### `js/views/meetings.js`

- **Upcoming** — `status = 'scheduled'` meetings whose effective status is not `completed` (`MeetingsAPI.fetchMeetings({ statusIn: ['scheduled'], effectiveCompleted: false })`).
- **My Meetings** — meetings where the caller is creator, organizer, or an internal participant (`MeetingsAPI.fetchMyMeetings()`, via the caller's own `meeting_participants` rows — `create_meeting` always inserts the creator as an organizer participant in the same transaction, so this single query already covers "creator" too). Drafts appear here with a visible **Draft** badge rather than in a separate tab, per this step's own "Drafts may instead appear in My Meetings" guidance.
- **Past** — `status = 'scheduled'` meetings whose effective status **is** `completed` (`end_at` in the past).
- **Cancelled** — `status = 'cancelled'` meetings.
- **Filters bar** (shared across all four tabs, applied client-side over the already-bounded, already-tab-scoped result set — matching `entry.js`/`requests.js`'s established search convention, and avoiding the `.or()`-with-user-text server-side filter-string risk `requests-api.js` itself documents and avoids): search (title/description substring), status, meeting type, visibility, location mode, date range, created-by-me. A separate "participant relationship" filter (listed as optional in this step's own instruction) was **not** added — the My Meetings tab already represents exactly that relationship, and a redundant filter duplicating a whole tab's own meaning was judged not worth the added surface.
- **Meeting list** — one shared row renderer used by all four tabs (the row shape is identical everywhere, so a single function is correct DRY, not premature abstraction). Each row shows title (+ creator, + Draft badge where applicable), type, date/start/end/timezone, status (icon **and** text, never color alone — `_statusLabel()`), location (room name via `activeBooking()`, external address, or a virtual indicator), and visibility. A `View` button opens the detail modal — matching `rooms.js`'s own precedent of keeping the list itself thin and putting every lifecycle action in the detail modal, referred to there as the "action menu."
- **Meeting details modal** — title, description (rendered via `textContent`/`_escapeHtml`, `white-space: pre-wrap` — never `innerHTML` of raw user text), meeting type, visibility, **both** the stored status and the derived effective status as two separate, clearly labeled fields, date/time/timezone, location detail (room booking status/time if linked, external address, or a safely-checked virtual link), creator, last-updated-by, participant list, attachments, cancellation detail (when cancelled), and created/updated timestamps. Contextual action bar: Edit (creator/supervisor/admin, non-cancelled, non-completed), Cancel (same authority, non-cancelled).
- **Create/Edit meeting form** — one shared function (fields overlap heavily; only status handling and the RPC called differ). Title/description/meeting type/visibility/date/start/end/timezone/location mode, with external-location/virtual-link fields shown only for their respective location mode. **No participant field and no room field at creation** — matches `create_meeting`'s own contract (participants only via `add_participant`; a room-mode meeting may exist with no booking yet). Client-side validation: title required (native), end-after-start, external-location-required-for-external, https-required-for-virtual — all mirrored, never replacing, the identical server-side checks in `create_meeting`/`update_meeting`. Submit button disables itself immediately to prevent duplicate submission; on success the modal closes and the active tab re-fetches (this codebase's established "success confirmation" idiom — no dedicated toast component exists anywhere in this app; every other module uses the same close-and-refresh pattern).
- **Edit rules enforced in the UI** (mirroring, never replacing, the RPC's own rules): a `scheduled` meeting's edit form shows no status field at all (nothing to select — `p_status` stays unset, so `update_meeting` leaves it unchanged; the server independently rejects `scheduled → draft` regardless). A `draft` meeting's edit form offers Draft/Scheduled. Cancelled meetings and meetings whose effective status is `completed` never show an Edit button at all. Time changes on a room-linked meeting are handled entirely server-side, atomically, by `update_meeting`'s own delegation to `reschedule_booking` — the frontend issues one `updateMeeting()` call and, on success, re-fetches both the meeting and its linked booking; on failure, the raised exception means **nothing** changed (the whole RPC call is one transaction, so step 1's meeting-field update and step 2's booking sync either both commit or neither does) — the error message shown is deliberately worded "was not updated / not changed," never implying a partial change.
- **Cancel meeting workflow** — confirmation modal with a reason field, mandatory only when the actor is not the creator (mirroring `cancel_meeting`'s own asymmetric rule, matching the identical UI convention `rooms.js` already established for `cancel_booking`'s own creator/manager asymmetry); an explicit warning is shown when an active linked booking exists ("will also be cancelled"). Refreshes the meeting detail and the active tab's list after success. Cancelled meetings are never removed from any tab's history — the Cancelled tab is exactly where they remain visible.
- **Participant management** — Internal (CorLink user picker, excluding already-active participants, sourced from `AdminAPI.listUsersByOrg(meeting.organization_id)` — the meeting's own org, not necessarily the viewer's, since a super admin may manage a meeting outside their own organization) and External (name required; email/phone/organization optional) tabs inside one Add Participant modal. **The role selector deliberately omits "Organizer"** — every meeting already has exactly one active organizer (the creator, auto-inserted by `create_meeting`), the sole-organizer uniqueness index makes a second organizer add always fail, and offering a control that can only ever fail would repeat the exact "don't expose a dead control" mistake this codebase's Rooms/Booking work already identified and corrected once (`docs/15` §1's conflict-override reality check) — so it was never introduced here in the first place. A same-normalized-email external participant already present is caught client-side before submit as a friendly pre-check; the server's own dedup index remains the actual, authoritative enforcement either way. Removal is a confirm-with-optional-reason modal calling `remove_participant` (soft removal server-side); the sole active organizer's row shows no Remove control at all (with a `title` explaining why), mirroring `remove_participant`'s own refusal.
- **Participant privacy** — `_renderParticipants` displays exactly what `meeting_participant_list()` returns: full contact fields for a privileged caller, `external_email`/`external_phone` rendered as "Email unavailable" for a non-privileged caller (never as blank/broken data, satisfying this step's own "gracefully render redacted values as unavailable" requirement). **Removed participants are not shown anywhere in this frontend, not even to managers** — `meeting_participant_list()` itself excludes `removed_at IS NOT NULL` rows entirely (confirmed by reading the function body directly), and this frontend never falls back to a raw-table query to recover that history (which would have meant querying `meeting_participants` directly for contact data beyond the caller's own row — exactly what this step's own instruction says not to do). This is a real, documented limitation, not an oversight — see §4.
- **Room assignment/change/detach** — Assign Room (shown whenever no active booking exists, regardless of current `location_mode`, matching `assign_room_booking`'s own unconditional `location_mode = 'room'` set on success — never invented as narrower than what the RPC actually allows) offers a room picker plus an advisory-only "Check Availability" button (`RoomsAPI.checkRoomAvailability`), then submits via `MeetingsAPI.assignRoomBooking`. Change Room (shown when an active booking exists) calls `RoomsAPI.rescheduleBooking` directly — the already-implemented, already-tested `reschedule_booking` RPC — with the meeting's own start/end/timezone and only the room changed, per this step's own "use the existing room assignment/reschedule behavior… do not invent direct booking-link updates" instruction; no new SQL or booking-link-update path was written. Detach Room calls `MeetingsAPI.detachRoomBooking`, with an explicit warning that the linked booking will be cancelled and the meeting will lose its room location. All three refresh the meeting detail modal afterward; Rooms module availability is checked via `AppShell.isModuleEnabled(user, 'rooms')` and the room-assignment UI is hidden with an explanatory note when Rooms is disabled for the org (the underlying RPCs would reject the attempt regardless — this is UX-only, matching every other permission mirror in this file).
- **Attachments** — reuses `entry-detail.js`'s existing `_renderAttachments`/dropzone/chip pattern verbatim (`record_type = 'meeting'`), not duplicated as a new component. Upload gated on `canManage(meeting) && status !== 'cancelled'` (matching `attachments_insert`'s own new `'meeting'` branch exactly). **No delete control exists anywhere in this view** — matching the fact that no view anywhere in this entire codebase exposes attachment deletion (confirmed by a codebase-wide search before writing this), despite `attachments_delete`'s uploader-only RLS policy existing at the database layer; this is a pre-existing, codebase-wide convention this step did not invent and is not expected to fix.
- **Notification bell integration** — `shell.js`'s notification click handler gained a second special case (`record_type === 'meeting'`, alongside the existing `meeting_room_booking` case) navigating to `#meetings` with a `meetingId` param; `meetings.js`'s `render()` opens that meeting's detail modal directly after the initial tab renders, mirroring `rooms.js`'s identical `bookingId`-param pattern exactly. Only the six notification types the migration actually ships are referenced anywhere in this frontend (indirectly — the frontend never constructs a notification message itself, since every Meetings notification is inserted server-side by the RPCs per `docs/12` §15): `meeting_created`, `participant_added`, `meeting_updated`, `room_assigned`, `meeting_cancelled`, `participant_removed`. No `meeting_reminder` type was invented or referenced anywhere, matching its explicit exclusion from the shipped migration.
- **Permission-aware UI** — `_canManage(meeting)` mirrors `can_manage_meeting()`: super admin, creator, or an org-wide supervisor/admin (the identical org-agnostic simplification `rooms.js`'s own `_isManagerOf` already established, safe because a meeting reachable via ordinary browsing is already same-org unless the viewer is a super admin, in which case `is_super_admin` already short-circuits). Every management control (Edit, Cancel, Add/Remove Participant, Assign/Change/Detach Room, Attachment upload) is gated by this mirror; none of it is the real security boundary — every server-side RPC/RLS check re-derives authorization independently and is the actual gate.

### Rooms/Booking integration — a single, minimal link (not a redesign)

Per this step's own "minimal integration link" allowance, `rooms.js`'s existing "Linked to a meeting" badge (previously display-only with no route, since the Meetings frontend did not exist) now renders as a clickable button navigating to `#meetings?meetingId=<id>` **only when** `AppShell.isModuleEnabled(user, 'meetings')` is true for the viewer — otherwise it remains the original, unchanged, non-clickable neutral badge. This is the only change made to `rooms.js`'s own workflows; no booking/block/room CRUD behavior, RLS mirror, or existing action was touched. Symmetrically, `meetings.js`'s own detail modal offers a "View in Rooms" button next to a linked booking's details, navigating to `#rooms?bookingId=<id>` — reusing Rooms's own already-shipped deep-link mechanism, adding nothing new to `rooms.js` itself.

### Hold workflow

Not applicable to Meetings — `create_booking_hold`/the `hold` status belong entirely to Rooms/Booking's own workflow (`docs/10`) and were already deliberately left out of `rooms.js`'s own UI (`docs/15` §1). `assign_room_booking` never creates a `hold`-status booking (it delegates only to `create_room_booking`/`submit_booking_request`), so nothing in this step needed to consider it.

## 2. Security review

- **No service-role key anywhere** — `meetings-api.js` only ever uses `getSupabase()` (the existing anon/authenticated client) and `Auth.getSession()`. Confirmed by grep on the finished file.
- **No direct writes to `meetings`/`meeting_participants`** — confirmed by reading the finished data-layer file: every `.from('meetings')`/`.from('meeting_participants')` call is a `.select()`; every mutation is a `.rpc()` call. Grepped for `.insert(`/`.update(`/`.delete(` against either table name — zero matches (only the file's own header comment mentions those words).
- **No client-supplied actor identity** — no RPC call anywhere passes a user id parameter as "who performed this action"; `add_participant`'s `p_user_id` is the target being added, not an actor override, exactly like every other target-identity parameter already used elsewhere in this codebase.
- **No production secrets added.**
- **Route guard verified** — `#meetings` is fail-closed exactly like every other module route (confirmed by reading `moduleGuardPasses()`; no special-casing was added for `meetings`); anonymous access denied at the RLS layer regardless (verified locally, §3).
- **Participant contact data uses the safe RPC** — `fetchMeetingParticipants` calls `meeting_participant_list()` exclusively; the one raw-table read in this file (`fetchMyMeetingIds`) is scoped to the caller's own row only, permitted by `meeting_participants_select`'s own policy, and never surfaces another participant's contact fields.
- **No cross-organization requests are intentionally made** — every org-scoped call (`RoomsAPI.fetchRooms`, `AdminAPI.listUsersByOrg`) uses `meeting.organization_id` explicitly (the meeting's own org, correct even for a super admin managing a meeting outside their own org), never a blanket or unscoped query.
- **No unsafe `innerHTML`** — every interpolation of a title, description, participant name, contact field, reason, or error message goes through `_escapeHtml()` (or, for the two spots that don't — `_uploadAttachments`'s failure-message join and the client-side search-filter comparison — both use `.textContent`/plain string comparison, never `innerHTML`, confirmed by direct inspection); verified by grepping every such interpolation in the finished file for a missing `_escapeHtml` wrapper.
- **Virtual URLs use safe handling** — a meeting's `virtual_link` is only ever rendered as a clickable `<a>` (with `target="_blank" rel="noopener noreferrer"`) after a client-side `^https:\/\//` regex check, defense-in-depth on top of the server's own identical, already-enforced `CHECK` constraint; a link that somehow failed the check (should never happen given the server constraint) is shown as a plain warning, never as a clickable/interpolated raw string.
- **No external participant portal behavior exists** — an external participant is displayed as a name/contact-info row only; nothing in this frontend creates a login surface, magic link, or any access path for them, matching `docs/12` §2's explicit exclusion.
- **No reminder logic, no recurring logic** — neither concept exists in the shipped schema or RPCs, and neither was referenced, simulated, or stubbed anywhere in this frontend.
- **No Meetings action bypasses a server RPC** — every mutation (create/update/cancel meeting, add/remove participant, assign/detach room) is a bare `.rpc()` call; the one Rooms RPC this file calls directly (`RoomsAPI.rescheduleBooking` for Change Room) is itself the already-implemented, already-tested `reschedule_booking` RPC, not a new write path.
- **No hosted Supabase access** — this entire step, like every step before it in this session, was conducted without touching either Supabase project.
- **MeetFlow untouched.**
- **No bare `alert()`** — grepped the finished file; zero matches.

**One honestly-flagged mock-harness gap:** `fetchMyMeetings()`'s combined `.or('created_by.eq.X,id.in.(...)')` PostgREST filter was **not** exercised by the mock (the hand-written mock's `.or()` stub is a no-op passthrough, so the My Meetings tab showed all 4 seeded meetings instead of the expected 3 in this specific mocked run — a mock limitation, not a defect in the shipped code). This was not independently verified against a live PostgREST instance this session, and is called out here rather than glossed over.

## 4. Known limitations

- **`fetchMyMeetings()`'s combined `.or()` filter was not verified against a live PostgREST instance this session** — see §3's explicit callout above.
- **Invitation status and attendance status cannot be edited from this frontend** — `add_participant` has no parameters for either field (they default to `'pending'`/`'unknown'` server-side), and no RPC exists anywhere in the shipped migration to update them post-add. This is a real gap in the current RPC set, not a frontend oversight; a future, separately-scoped migration step would need to add such an RPC before this frontend (or any frontend) could expose that control.
- **Removed participants are never shown, to anyone, anywhere in this frontend** — `meeting_participant_list()` itself excludes them, and this frontend does not fall back to a raw-table query to recover that history (which would require reading other participants' rows beyond what `docs/12` §13's privacy design permits this frontend to do). A manager wanting to see removal history today has no UI path to it at all.
- **No attachment delete control exists** — matching the pre-existing, codebase-wide convention (confirmed absent in every other module too, not unique to Meetings).
- **`checkRoomAvailability` is advisory, not a lock** — identical, already-documented limitation inherited from Rooms/Booking (`docs/15` §4); a genuine narrow TOCTOU window exists between the check and the actual `assignRoomBooking` submit, and the RPC itself is the real, re-validating authority.
- **No live hosted-Supabase verification** — this entire step, like every database/frontend phase before it, was conducted without touching either Supabase project. Outstanding hosted-project checks: apply `patch-meetings-foundation.sql` (if not already applied there) and `patch-meetings-route-activation.sql`; confirm an org admin can enable Meetings from `admin.js`'s Modules tab once the route is set; re-run this step's local RPC test matrix against the hosted project to confirm identical behavior; verify the `fetchMyMeetings()` combined `.or()` filter against real PostgREST (the one gap called out above).
